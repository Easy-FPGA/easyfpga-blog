# Vitis C Code

Vitis is the AMD/Xilinx IDE for embedded software targeting MicroBlaze, Zynq ARM, and Versal platforms. Once the hardware design is exported as an `.xsa` file, Vitis generates a Board Support Package (BSP) that maps hardware registers to C constants and provides low-level driver functions. This episode walks through writing the UART loopback application step by step.

## Vivado → Vitis Workflow

```
Vivado
  └── Generate Bitstream
  └── Export Hardware (.xsa, include bitstream)
         │
         ▼
Vitis
  1. File → New Platform Project
     - Select .xsa file
     - BSP is generated automatically (xparameters.h, drivers)
  2. File → New Application Project
     - Select the platform
     - Template: Empty Application (C)
  3. Write application C code
  4. Build All
  5. Run → Launch on Hardware (JTAG)
```

## Key BSP Header: xparameters.h

The Board Support Package auto-generates hardware addresses and device IDs:

```c
// Auto-generated by Vitis BSP — do not edit manually
#define XPAR_UARTLITE_0_BASEADDR      0x40600000UL
#define XPAR_UARTLITE_0_DEVICE_ID     0
#define XPAR_UARTLITE_0_BAUDRATE      115200
#define XPAR_UARTLITE_0_USE_PARITY    1
#define XPAR_UARTLITE_0_ODD_PARITY    0

#define XPAR_CPU_CORE_CLOCK_FREQ_HZ   100000000UL
```

> **Never hardcode addresses** in application code — always use `XPAR_*` macros. If you re-export the `.xsa` with a different address assignment and rebuild the BSP, the macro updates automatically and your application requires no changes. This is the portability contract between Vivado hardware changes and Vitis C code.

## Potential Issue: BRAM Size Overflow

If the C application is too large, the linker will fail:

```
ld: region `lmb_bram_memory' overflowed by 4096 bytes
```

**Fix**: Increase BRAM size in the Block Design

```
In Vivado Block Design:
1. Double-click local memory BRAM
2. Change "Size (KB)": 32 → 64 or 128
3. Re-synthesize and re-export .xsa
```

A minimal C application (BSP + loopback) fits in 32 KB. Adding `printf`, FreeRTOS, or a network stack can quickly exceed this — upgrade to 64 or 128 KB BRAM for anything beyond basic demos.

## Loopback C Code

```c
#include "xparameters.h"
#include "xuartlite.h"

#define UART_DEVICE_ID   XPAR_UARTLITE_0_DEVICE_ID

int main(void) {
    XUartLite uart;
    u8 recv_buf[1];
    u8 send_buf[1];

    // Initialize UART Lite
    XUartLite_Initialize(&uart, UART_DEVICE_ID);

    while (1) {
        // Wait for a byte to arrive
        while (XUartLite_Recv(&uart, recv_buf, 1) == 0);

        // Echo it back
        send_buf[0] = recv_buf[0];
        XUartLite_Send(&uart, send_buf, 1);
    }

    return 0;
}
```

> **`XUartLite_Recv` return value**: The function returns the number of bytes actually received (0 or 1). Polling `while (... == 0)` spins until at least one byte arrives. This is a busy-wait loop — the CPU is fully occupied. Acceptable for simple demos; use interrupt-driven receive for production code.

## Printf via UART

Enable stdin/stdout redirection to UART in BSP settings:

```
Vitis → Platform Project → Board Support Package → Modify BSP Settings
  → standalone:
      stdin  = uartlite_0
      stdout = uartlite_0
```

Then use `printf` normally:

```c
#include <stdio.h>
#include "xparameters.h"

int main(void) {
    printf("MicroBlaze UART Test\n");
    printf("System clock: %lu Hz\n", XPAR_CPU_CORE_CLOCK_FREQ_HZ);

    while (1) {
        u8 ch = XUartLite_RecvByte(XPAR_UARTLITE_0_BASEADDR);
        printf("Received: 0x%02X ('%c')\n", ch, ch);
        XUartLite_SendByte(XPAR_UARTLITE_0_BASEADDR, ch);  // echo
    }
}
```

> **`printf` overhead**: Each `printf` call serializes characters to the UART FIFO one at a time. At 115,200 bps, printing a 20-character string takes approximately 1.7 ms — during which the CPU is blocked waiting for the FIFO. For time-sensitive data paths, use formatted output buffers or a lightweight logging mechanism.

## Interrupt-Driven UART (Advanced)

```c
// Interrupt handler — called by XIntc when UART receives data
void uart_rx_handler(void *ref, unsigned int count) {
    XUartLite *uart = (XUartLite *)ref;
    u8 buf[16];
    XUartLite_Recv(uart, buf, count);
    // Process received data here
}

// Setup
XUartLite_SetRecvHandler(&uart, uart_rx_handler, &uart);
XUartLite_EnableInterrupt(&uart);
// Connect to MicroBlaze interrupt controller (XIntc)
```

Interrupt-driven receive allows the CPU to execute other tasks between byte arrivals and only wake up when data appears. This is the standard pattern for all production embedded UART code.

## Key Takeaways

- Always use `XPAR_*` macros for addresses — never hardcode `0x40600000` in application source; the BSP regenerates macros automatically when hardware changes
- BRAM overflow at link time (`region lmb_bram_memory overflowed`) means the binary exceeds the available memory — increase BRAM depth in the Block Design
- Polling with `XUartLite_Recv` is acceptable for simple demos; `XUartLite_SetRecvHandler` + `XUartLite_EnableInterrupt` is required for responsive, production-quality applications
- Setting `stdout = uartlite_0` in BSP settings routes all `printf` output to the UART — enabling the most convenient and familiar debug channel available on bare-metal
